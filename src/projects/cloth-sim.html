Cloth Simulation with Verlet Integration

A deep dive into building a real-time cloth simulation from scratch — covering constraint solving, structural vs. shear springs, and the surprising elegance of position-based dynamics.

Open Interactive Simulation

Cloth simulation has been a staple of game engines and VFX pipelines for decades, yet the core ideas behind it are remarkably accessible. At its heart, a piece of cloth is just a grid of point masses connected by springs — and the challenge is making those springs behave in a way that feels convincingly physical without blowing up your frame budget.

In this article I'll walk through the simulation I built, explain the key algorithmic decisions, and show you where the math actually lives in the code. By the end you should have a solid mental model for building your own.

Why Verlet Integration?

Most introductory physics tutorials reach for Euler integration — update velocity, then update position. It's simple and intuitive, but it has a dirty secret: it adds energy to the system. Springs become stiffer over time, oscillations grow, and eventually your simulation explodes.

Verlet integration sidesteps this by never storing velocity explicitly. Instead, velocity is implicit in the difference between the current and previous position. This makes it naturally energy-conservative and beautifully stable even at larger timesteps.

xn+1 = 2xn − xn−1 + a·Δt²
Note Verlet integration is second-order accurate — the same order as Runge-Kutta 2 — but requires no velocity storage and is trivially symplectic, making it ideal for constraint-heavy simulations.

Implementation

Here's the core particle update loop. Each particle stores its current position and its previous position; that's all we need to integrate forward in time.

JavaScript 
class Particle {
  constructor(x, y) {
    this.pos  = { x, y };
    this.prev = { x, y };  // previous position for Verlet
    this.pinned = false;
  }

  update(dt) {
    if (this.pinned) return;

    const ax = 0;
    const ay = GRAVITY;   // e.g. 980 px/s²

    const vx = this.pos.x - this.prev.x;
    const vy = this.pos.y - this.prev.y;

    this.prev.x = this.pos.x;
    this.prev.y = this.pos.y;

    this.pos.x += vx * DAMPING + ax * dt * dt;
    this.pos.y += vy * DAMPING + ay * dt * dt;
  }
}

The DAMPING factor (typically 0.98–0.995) bleeds off a small amount of energy each frame, simulating air resistance and preventing the simulation from becoming a perpetual-motion machine.

Building the Cloth Grid

The cloth itself is a 2D grid of particles. Each particle is connected to its immediate neighbours — up, down, left, right — with structural springs. We also add shear springs along the diagonals to prevent the cloth from collapsing into an accordion fold, and optionally bend springs that skip one particle to resist folding.

JavaScript 
function buildCloth(cols, rows, spacing) {
  const particles = [];
  const constraints = [];

  // Create particle grid
  for (let r = 0; r < rows; r++) {
    for (let c = 0; c < cols; c++) {
      const p = new Particle(c * spacing, r * spacing);
      if (r === 0) p.pinned = true;  // pin top row
      particles.push(p);
    }
  }

  const idx = (r, c) => r * cols + c;

  // Structural + shear + bend springs
  for (let r = 0; r < rows; r++) {
    for (let c = 0; c < cols; c++) {
      if (c + 1 < cols)  addConstraint(constraints, particles, idx(r,c), idx(r,c+1));   // structural →
      if (r + 1 < rows)  addConstraint(constraints, particles, idx(r,c), idx(r+1,c));   // structural ↓
      if (r + 1 < rows && c + 1 < cols) addConstraint(constraints, particles, idx(r,c), idx(r+1,c+1)); // shear ↘
      if (r + 1 < rows && c - 1 >= 0)  addConstraint(constraints, particles, idx(r,c), idx(r+1,c-1)); // shear ↙
      if (c + 2 < cols)  addConstraint(constraints, particles, idx(r,c), idx(r,c+2));   // bend →→
      if (r + 2 < rows)  addConstraint(constraints, particles, idx(r,c), idx(r+2,c));   // bend ↓↓
    }
  }

  return { particles, constraints };
}

Constraint Solving

Springs in a cloth simulation aren't governed by Hooke's law in the traditional sense. Instead, we treat each spring as a distance constraint — a rule that says two particles must be exactly d units apart. We enforce this iteratively by projecting the particles back onto the constraint surface.

Running this solver multiple times per frame (typically 5–20 iterations) is what gives the cloth its stiffness without requiring a stiff ODE solver. More iterations = stiffer cloth = more stable, but at higher CPU cost.

JavaScript 
function satisfyConstraint(c) {
  const dx = c.b.pos.x - c.a.pos.x;
  const dy = c.b.pos.y - c.a.pos.y;
  const dist  = Math.sqrt(dx*dx + dy*dy) || 0.0001;
  const diff  = (dist - c.restLength) / dist;

  // Correction vector, split 50/50 between both particles
  const corrX = dx * diff * 0.5;
  const corrY = dy * diff * 0.5;

  if (!c.a.pinned) { c.a.pos.x += corrX; c.a.pos.y += corrY; }
  if (!c.b.pinned) { c.b.pos.x -= corrX; c.b.pos.y -= corrY; }
}

// Run N iterations per frame
for (let i = 0; i < SOLVER_ITERATIONS; i++) {
  constraints.forEach(satisfyConstraint);
}

Tearing & Interaction

One of the most satisfying additions is cloth tearing. When a constraint is stretched beyond a threshold, we simply remove it from the list. Because the solver is purely position-based, this requires no special handling — the cloth naturally falls apart.

Mouse interaction follows the same principle. On mousedown, we find the nearest particle within a grab radius, pin it to the cursor, and drag it through the simulation. On mouseup, we release the pin — and optionally tear every constraint attached to that particle if the cursor was moving fast enough.

Performance Considerations

A 40×40 cloth grid has 1,600 particles and roughly 9,000 constraints. Running 10 solver iterations means 90,000 constraint projections per frame. In JavaScript this is fast enough to hit 60 fps comfortably — but there's still room to optimise.

Spatial Partitioning

For collision detection (cloth against itself or against scene objects), a flat O(n²) search is untenable. A uniform spatial hash — where each particle is binned into a 2D cell proportional to the maximum constraint rest length — reduces average lookup cost to O(1), making self-collision feasible in real-time.

Tip Typed arrays (Float32Array) for particle positions and a flat constraint array encoded as index pairs can yield a 2–3× throughput improvement over a class-based approach in V8, mainly due to better cache locality.
